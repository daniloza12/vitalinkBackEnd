● The fix: /{path:^(?!api).*$}/**/{path:[^\\.]*} → /**/{path:[^\\.]*}.

  Spring's PathPatternParser (default since 2.6) forbids any segment after **.
  The replacement /**/{path:[^\\.]*} still catches all nested SPA routes while
  the [^\\.]* regex excludes static files (anything with a dot). Your actual API
   routes under /api/v1/... will still match their own controllers first due to
  Spring's specificity-based route ordering.

  Rebuild and redeploy to Azure.

> que pasa si no uso CorsConfig 

● Si no tienes CorsConfig, Spring Boot no envía los headers
  Access-Control-Allow-Origin en las respuestas.

  Resultado: el navegador bloquea todas las peticiones del frontend hacia tu
  API. Verías este error en la consola del navegador:

  Access to XMLHttpRequest at 'https://tu-api.azurewebsites.net/api/v1/...'
  from origin 'https://tu-frontend.com' has been blocked by CORS policy:
  No 'Access-Control-Allow-Origin' header is present on the requested resource.

  Las peticiones desde Postman, curl o cualquier cliente no-browser seguirían
  funcionando sin problema, porque CORS es una restricción del navegador, no del
   servidor.

> y si uso asi con * en allowedOriginPatterns 

● Con allowedOriginPatterns("*") el servidor responde con:

  Access-Control-Allow-Origin: https://tu-frontend.com

  Acepta peticiones desde cualquier dominio — localhost, Azure Static Apps,
  Cloudflare Workers, cualquier cliente browser. Es lo más permisivo posible.

  Para un proyecto en desarrollo/educativo como VitaLink está bien. En
  producción real se recomendaría listar solo los dominios del frontend para
  evitar que cualquier sitio web consuma tu API.
